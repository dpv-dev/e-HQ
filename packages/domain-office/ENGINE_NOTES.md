# Office P&L Engine Notes

This package computes the Office / EOF P&L read surfaces from rows matching the
clean B1 schema. It is fixture-tested only in this phase; live parity against
PHP comes after the ETL phase lands data in the target schema.

Audit fixes encoded here:

1. BUG-M1 fixed: project P&L and partner P&L without a department filter read
   the ledger transaction rows directly and return non-zero totals on the
   fixture dataset.
2. Division single source: by-division reporting derives from
   `category.division_id`; there is no denormalized division name path.
3. Department grain: by-department and department P&L read
   `financial_allocations`. Division/category reads use ledger transactions.
   Department totals and division totals can intentionally differ for
   shared-cost transactions.

## Office Allocation And Validation Notes

This package also computes the Office / EOF write-side allocation decisions as
pure plans. The API layer can persist those plans inside one transaction later;
this phase does not connect to a database.

Rules encoded here:

1. Validation fails loud with stable reasons for invalid amount, missing FX,
   missing VAT amount, missing allocations, or incomplete allocations.
2. Client-supplied allocation sets keep the EOF tolerance: basis points must be
   within 1 bp of 10000 and amounts within 1 minor unit of the transaction.
3. Shared-cost expansion is exact when generated by the engine:
   `splitRemainderLast` creates one allocation per active rule and the final
   amount sum equals the transaction amount exactly.
4. Active shared-cost rules may use the EOF 1 bp tolerance; the last rule is
   adjusted by that tolerance before calling `splitRemainderLast`, so the shared
   primitive remains the single split path.
5. No `division_name` is written. Allocations target real department ids only.
6. Project department auto-add is an explicit plan output with an info alert,
   not a hidden write.

## Office B2 ETL Notes

The B2 transform is pure and local: it accepts rows already extracted from the
confirmed MySQL source and never opens a connection. The live source contract is
`u384688932_HZ0LD` with prefix `wp_`; Office tables are `wp_eof_*`.

Guards encoded here:

1. Phase-0 source counts must match before any transform starts: transactions
   3107, financial allocations 2832, categories 551, departments 93, partners
   247, projects 12, bank raw transactions 3093, and bank reconciliations 1568.
2. The clean dimensional path drops legacy `financial_allocations.division_name`
   completely. The confirmed drift count is 1760 and is expected; division is
   derived only through `category.division_id`.
3. Golden-master parity is exact on validated active rows: 2396 transactions,
   income 2214542460 cents, expense 1362642716 cents. The ETL asserts both the
   raw transformed rows and `domain-office` global P&L against these values.
4. Project and partner P&L without a department filter intentionally diverge
   from legacy WordPress BUG-M1: WordPress reports zero, the new engine returns
   the actual ledger total.
