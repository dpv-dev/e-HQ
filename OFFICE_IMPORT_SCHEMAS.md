# Office Import Schemas

Reference for Office-side imports handled through `eof/v1`. This file documents
observed source formats so the API preview/import workflow can be adapted to the
real files without moving bank logic into the UI.

## Safe Workflow

- Office owns bank statements, cashflow, receipts, invoices, and reconciliation
  imports.
- Distribution owns distributor royalty exports only.
- Imports use `preview` -> `confirm` with an `Idempotency-Key`.
- Confirmed source rows are batch-based and reversible.
- Rows that cannot be classified go to Pending with the exact reason and fix
  path; the UI never guesses categories or mutates source rows in place.
- Money is stored as integer micro-units in the source currency. No floats.

## MCB Current Account PDF

Observed sample:

- File: `Euro MCB Current Account statement (5).pdf`
- Bank format: MCB current account statement PDF
- Pages: `1`
- Currency: `EUR`
- Statement period: `04/05/2026 to 29/05/2026`
- Account display: `MCB EUR current ****4509`
- IBAN/account numbers should be masked in UI.
- Opening balance: `0.00 EUR`
- Closing balance: `0.00 EUR`
- Transaction starts detected: `9`

Text structure:

- Header contains account metadata, statement period, currency, IBAN, customer
  address, and MCB warning/legal copy.
- Transaction table columns are visually:
  `TRANS DATE`, `VALUE DATE`, `TRANSACTION DETAILS`, `CREDIT`, `DEBIT`,
  `BALANCE`.
- Extracted text folds amount, balance, and details into a single line after
  the two dates.
- Details can continue onto following lines until the next date-led row.

Parsing boundary:

- The API parser should infer debit/credit from the server-side parsed columns
  or balance movement. The app must not compute debit/credit itself.
- Details such as transfer references and counterparties remain source text and
  can later support reconciliation suggestions.

Suggested idempotency fingerprint:

```text
mcb:{masked-account-or-account-hash}:{statement-start}:{statement-end}:{file-hash}
```

## MUR Transaction History PDF

Observed sample:

- File: `bankStatement_1Oct2024 to 28Jan2025.pdf`
- Bank format: transaction-history PDF generated by JasperReports
- Pages: `14`
- Currency: `MUR`
- Statement period: `01/10/2024 to 28/01/2025`
- Account display: `MUR statement ****5101`
- Transaction starts detected: `327`
- Opening-side balance observed from the oldest visible row: `275,745.27 MUR`
- Closing/available balance: `901,726.55 MUR`

Text structure:

- Header contains account statement metadata, branch, currency, date range, and
  page number.
- Transaction table columns are:
  `Date`, `Instrument ID`, `Debit`, `Credit`, `Balance`, `Particulars`.
- Extracted rows start with dates like `23,Jan,2025`.
- Particulars and transaction IDs wrap to continuation lines, often with
  `ID:...`.
- Statement is ordered newest first in the sample.

Parsing boundary:

- The parser should group continuation lines under the latest date-led
  transaction.
- Debit/credit column assignment is a server parser responsibility.
- Duplicate detection should use account hash, date, amount, balance, reference,
  and normalized particulars.

Suggested idempotency fingerprint:

```text
bank-statement:{account-hash}:{statement-start}:{statement-end}:{file-hash}
```

## Cashflow Workbook

Observed sample:

- File: `Cashflow 2025.xlsx`
- Workbook sheets: `REPORT`, `BOOKING FEE`, `P&L PER EVENT 2025`, month sheets,
  `PENDING`, `invoices_wip`, `Payout 2024`, `CONTACTS`, and expense extracts.
- Monthly sheets include Dec 2024, Jan-Dec 2025, and Jan-Apr 2026 samples.
- Main monthly columns include:
  `DATE ISSUED`, `INVOICE REF`, `CLIENT / EXPENSES`, `AMOUNT MUR`,
  `PAID ON (TRANSACTION)`, `AMOUNT PAID`, `STILL DUE`,
  `SBI CLOSING BALANCE`, `REMARKS`.
- Some month sheets add `VAT 15%`.
- Candidate rows with a client/expense and amount across observed monthly tabs:
  `1033`.
- REPORT sheet contains monthly/quarterly inflow, outflow, and net flow
  formulas.

Parsing boundary:

- Workbook rows are supporting source/projection records, not canonical ledger
  facts until confirmed.
- Formulas must not be used as canonical money facts without preserving the raw
  source values and audit trail.
- Corrections later become audited override records, not edits to imported
  rows.

Suggested idempotency fingerprint:

```text
cashflow-workbook:{workbook-year-or-range}:{file-hash}
```

## Receipt And Invoice PDFs

The generic `pdf` import lane is for supplier receipts and invoices, not bank
statements when the source is known. It should detect:

- invoice/receipt date,
- supplier or customer name,
- reference number,
- currency,
- total amount,
- VAT/tax lines when present,
- candidate department/division/category only as a suggestion.

Rows that cannot be safely classified go to Pending.
